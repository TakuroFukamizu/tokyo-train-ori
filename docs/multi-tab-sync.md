# マルチタブ同期による路線表示分割仕様

## 概要

BroadcastChannel API を使い、同一オリジンで開かれているタブの数に応じて、各タブに表示する路線・駅を自動的に分割する。複数のモニターやウィンドウを並べて、それぞれ異なるレイヤーの路線網を同時に俯瞰できる。

## 表示分割ルール

| タブ数 | タブ 0 | タブ 1 | タブ 2 | タブ 3+ |
|--------|--------|--------|--------|---------|
| 1 | 全路線・全駅 | - | - | - |
| 2 | 23区内完結路線 | 接続路線 | - | - |
| 3+ | 地下鉄 (Metro/都営) | JR・私鉄 (23区内) | 接続路線 | 全路線 (フォールバック) |

### 路線分類の定義

| カテゴリ | 条件 | 例 |
|----------|------|----|
| `subway` | `operator_code` が `TokyoMetro` または `Toei` で、全駅が23区 bbox 内 | 銀座線、丸ノ内線、都営大江戸線 |
| `inner` | 全駅が23区 bbox 内だが地下鉄ではない | 山手線、京浜東北線、りんかい線、ゆりかもめ |
| `connecting` | 23区 bbox の内外両方に駅がある路線 | 京王線、中央線快速、西武新宿線、小田急小田原線 |

23区バウンディングボックス:
- 緯度: 35.53 - 35.82
- 経度: 139.56 - 139.92

全駅が bbox 外の路線（青梅線、南武線など）は現在描画対象外。

## タブインデックスの決定

各タブは起動時に `crypto.randomUUID()` で固有IDを生成する。全アクティブタブのIDを辞書順ソートし、その中での自分の位置がタブインデックスとなる。

- 最初に開いたタブが必ずインデックス 0 になるわけではない（UUID のソート順による）
- タブの開閉により全タブのインデックスが再計算され、表示内容が動的に切り替わる

## 通信プロトコル

BroadcastChannel 名: `"ori-tab-sync"`

| メッセージ | タイミング | 目的 |
|------------|-----------|------|
| `{ type: "join", id }` | タブ起動時 | 新規参加を通知 |
| `{ type: "announce", id }` | join 受信時 | 既存タブが新規タブに自分の存在を返答 |
| `{ type: "heartbeat", id }` | 2秒間隔 | 生存確認 |
| `{ type: "leave", id }` | `beforeunload` 時 | 離脱を通知 |

タイムアウト: 4秒間ハートビートがないタブは自動除外される。

## ファイル構成

| ファイル | 役割 |
|----------|------|
| `src/tabSync.ts` | BroadcastChannel によるタブ間同期。`TabSync` クラスを export |
| `src/stationFilter.ts` | 路線分類 (`classifyLines`) とフィルタリング (`filterStations`, `getCategoriesForTab`) |
| `src/stationRenderer.ts` | `fetchStations` (キャッシュ付きデータ取得) と `renderStations` (再描画対応) |
| `src/main.ts` | `TabSync` を初期化し、タブ構成変化時に駅・電車を再描画 |

## 再描画の仕組み

タブ構成が変化すると以下が順に実行される:

1. `getCategoriesForTab(tabCount, tabIndex)` で表示すべきカテゴリを決定
2. `filterStations()` で全駅データからカテゴリに該当する駅のみ抽出
3. `renderStations()` が既存の描画グループ (`"station-render-group"`) を破棄し、フィルタ済みデータで再構築
4. `trainRenderer.load()` で該当路線の電車シミュレーションも再構築

Three.js のジオメトリ・マテリアルは再描画時に `dispose()` されるため、メモリリークは発生しない。

### 列車シミュレーションの再構築

`TrainRenderer.load()` は複数回呼び出しに対応しており、タブ変更時に以下を行う:

1. **アクティブ列車の回収**: 走行中の全列車メッシュを非表示にし、オブジェクトプールに返却
2. **路線インデックスの再構築**: `lineMap` をクリアし、フィルタ済みの `RailLine[]` で再構築
3. **時刻表データのキャッシュ**: 全路線の時刻表は初回のみ fetch し、以降はキャッシュを使用
4. **コーンプールの再利用**: メッシュプールは初回のみ確保し、タブ変更時は再利用

これにより、`update()` ループは新しい `lineMap` に含まれる路線の列車のみをスポーンする。時刻表に路線が存在しても `lineMap` にマッチしなければスキップされるため、タブごとに該当路線の列車だけが表示される。
